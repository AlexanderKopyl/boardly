# Issue #42 Plan: Create Frontend Next.js TypeScript Skeleton

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development
> (recommended) or superpowers:executing-plans to implement this plan task-by-task.
> Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Create the initial Boardly frontend skeleton under `frontend/` with Next.js 16,
TypeScript 6, and context-based hexagonal architecture ready for the IdentityAccess
authentication implementation.

**Architecture:** Manual initialization of Next.js App Router with strict TypeScript under
`frontend/src/`. Source organized into `app/` (routing), `shared/` (cross-context primitives),
and `contexts/identity-access/` (first product context with domain, application,
infrastructure, and presentation layers). No real authentication logic — stubs and type
definitions only.

**Tech Stack:** Next.js 16.2.6, React 19.2.6, TypeScript 6.0.3, ESLint 10.3.0, npm

## 6. Shared Layer Plan

### Task 10 — src/shared/config/env.ts

Reads and exports typed environment config. Only `NEXT_PUBLIC_API_URL` is needed
for this skeleton:

```ts
export const API_URL = process.env.NEXT_PUBLIC_API_URL ?? ''
```

No secrets. No tokens. No other variables.

### Task 11 — src/shared/api/api-error.ts

Normalized API error representation. Must expose:

- `statusCode: number` — HTTP response status
- `errorCode: string` — stable backend error code string (drives UI behavior)
- `message: string` — display message

Logic must use `errorCode`, never raw `message` strings.

### Task 12 — src/shared/api/http-client.ts

Shared fetch wrapper. Responsibilities:

- Base URL from `shared/config/env.ts`
- `credentials: 'include'` on every request (enables HttpOnly refresh cookie)
- `Authorization: Bearer <token>` injection when an access token is provided
- JSON serialization and deserialization
- Error normalization into `ApiError`
- 401 refresh retry hook — accepts an optional `onUnauthorized` callback to trigger
  `refreshSessionUseCase` and retry the original request once
- No infinite refresh loop: retry only once per original request

Does not store tokens. Does not know about auth context or React.

### Task 13 — src/shared/ui/Button.tsx + src/shared/ui/Input.tsx

Generic, minimally styled components. No auth logic.

`Button` props: `type`, `disabled`, `onClick`, `children`.
`Input` props: `type`, `value`, `onChange`, `placeholder`, `disabled`, `required`.

## 7. IdentityAccess Context Plan

### Task 14 — domain files

**account-status.ts**

```ts
export type AccountStatus = 'pending_approval' | 'active' | 'rejected' | 'disabled'
```

**account.ts**

```ts
export interface Account {
  id: string
  email: string
  name: string
  status: AccountStatus
}
```

**auth-session.ts**

```ts
export interface AuthSession {
  accessToken: string
  expiresAt: Date
  account: Account
}
```

**auth-error.ts**

```ts
export type AuthErrorCode =
  | 'invalid_credentials'
  | 'account_not_active'
  | 'too_many_login_attempts'
  | 'invalid_refresh_token'

export class AuthError extends Error {
  constructor(public readonly code: AuthErrorCode, message?: string) {
    super(message ?? code)
  }
}
```

No React imports. No Next.js imports. No HTTP imports. No browser storage access.

### Task 15 — application/ports/auth-gateway.ts

```ts
export interface LoginResult {
  accessToken: string
  expiresIn: number
}

export interface RegisterResult {
  id: string
  email: string
  name: string
  status: 'pending_approval'
}

export interface AuthGateway {
  login(email: string, password: string): Promise<LoginResult>
  register(email: string, password: string, name: string): Promise<RegisterResult>
  refreshSession(): Promise<LoginResult>
  logout(): Promise<void>
  getMe(accessToken: string): Promise<Account>
}
```

No React. No HTTP client. Interface only.

### Task 15 — application/ports/auth-session-store.ts

```ts
export interface AuthSessionStore {
  save(session: AuthSession): void
  get(): AuthSession | null
  clear(): void
}
```

No React. No HTTP client. Interface only.

### Task 16 — application/use-cases/login.ts

Signature: `loginUseCase(input, deps) => Promise<AuthSession>`

Flow:
1. Call `deps.gateway.login(email, password)`.
2. Build `AuthSession` from `LoginResult` (`expiresAt = now + expiresIn * 1000`).
3. Call `deps.store.save(session)`.
4. Return session.

Depends only on `AuthGateway` and `AuthSessionStore` ports.

### Task 16 — application/use-cases/register.ts

Signature: `registerUseCase(input, deps) => Promise<RegisterResult>`

Flow:
1. Call `deps.gateway.register(email, password, name)`.
2. Return result.

Does not issue tokens. Returns `pending_approval` account. No session is created.

### Task 16 — application/use-cases/refresh-session.ts

Signature: `refreshSessionUseCase(deps) => Promise<AuthSession>`

Flow:
1. Call `deps.gateway.refreshSession()`.
2. Build `AuthSession` from result.
3. Call `deps.store.save(session)`.
4. Return session.

### Task 16 — application/use-cases/logout.ts

Signature: `logoutUseCase(deps) => Promise<void>`

Flow:
1. Call `deps.gateway.logout()`.
2. Call `deps.store.clear()`.

### Task 16 — application/use-cases/bootstrap-session.ts

Signature: `bootstrapSessionUseCase(deps) => Promise<AuthSession | null>`

Flow:
1. Call `refreshSessionUseCase(deps)`.
2. On success: return `AuthSession`.
3. On `AuthError` with code `invalid_refresh_token`: call `deps.store.clear()`, return `null`.
4. Other errors propagate.

Called on app mount from `providers.tsx`. This is the only point where the frontend
attempts to restore a session after a page reload.

### Task 17 — infrastructure/http/auth-api-contracts.ts

TypeScript interfaces matching ADR-0005 backend API response shapes:

```ts
export interface LoginResponse {
  accessToken: string
  tokenType: 'Bearer'
  expiresIn: number
}

export interface RegisterResponse {
  id: string
  email: string
  name: string
  status: 'pending_approval'
}

export interface MeResponse {
  id: string
  email: string
  name: string
}
```

No `refresh_token` fields — the refresh token is cookie-only and never returned in JSON.

### Task 17 — infrastructure/http/auth-http-gateway.ts

Implements `AuthGateway`. Uses `shared/api/http-client.ts`.

Endpoints:

| Method | Path | Notes |
|--------|------|-------|
| `POST` | `/api/auth/login` | Returns `LoginResponse` |
| `POST` | `/api/auth/register` | Returns `RegisterResponse` |
| `POST` | `/api/auth/refresh` | `X-CSRF-Intent: auth-refresh` header required |
| `POST` | `/api/auth/logout` | `X-CSRF-Intent: auth-refresh` header required |
| `GET` | `/api/auth/me` | `Authorization: Bearer` header required |

Maps backend responses to domain model types. All requests include `credentials: 'include'`.

### Task 18 — infrastructure/state/auth-memory-store.ts

Implements `AuthSessionStore`. Uses a module-level variable:

```ts
let session: AuthSession | null = null
```

`save`, `get`, `clear` operate on this variable. No `localStorage`, `sessionStorage`,
`IndexedDB`, or cookies used.

### Task 19 — presentation/hooks/useAuth.ts

React hook wrapping auth use cases. Provides:

- `session: AuthSession | null`
- `isLoading: boolean`
- `login(email: string, password: string): Promise<void>`
- `register(email: string, password: string, name: string): Promise<void>`
- `logout(): Promise<void>`

Reads from and writes to auth context (provided by `providers.tsx`). Does not call
backend directly.

### Task 20 — presentation/ui/LoginForm.tsx

Client component (`'use client'`). Form with email + password fields. Uses
`shared/ui/Button` and `shared/ui/Input`. Calls `useAuth().login(...)`. Shows
`AuthError` on failure. Redirects to `/dashboard` on success.

### Task 20 — presentation/ui/RegisterForm.tsx

Client component (`'use client'`). Form with email, password, and name fields.
Calls `useAuth().register(...)`. Shows `pending_approval` confirmation message
after successful submit. Does not redirect to dashboard (no session issued).

### Task 20 — presentation/ui/LogoutButton.tsx

Client component (`'use client'`). Button that calls `useAuth().logout()`.
Redirects to `/login` after logout completes.

### Task 21 — presentation/guards/ProtectedRoute.tsx

Client component (`'use client'`). Checks session state:

- `isLoading = true`: Renders a loading indicator.
- `session = null` after loading: Redirects to `/login`.
- `session != null`: Renders `{children}`.

Used in `src/app/dashboard/page.tsx`.

## 11. Non-Goals

- Real login, register, refresh, or logout API behavior.
- Token persistence of any kind.
- JWT decoding or validation on the frontend.
- Component library (Tailwind, MUI, Radix UI, shadcn/ui, etc.).
- Data-fetching library (TanStack Query, SWR, etc.).
- State management library (Zustand, Redux Toolkit, etc.).
- Form validation library.
- API client code generation from OpenAPI.
- ESLint import boundary rules.
- Unit or integration tests.
- Project, Issue, Board, Workflow, or Reporting frontend contexts.
- Admin UI or admin approval flow.
- Backend API changes.
- Docker, CI/CD, or deployment configuration.
- Real-time updates (SSE, WebSockets, Mercure).
- Password reset flow.
